# 环境配置和技术栈选型

# Android 蓝牙锄大地扑克游戏项目开发环境与工程初始化说明（Kotlin 方案）

## 1\. 文档目的

本文档用于说明 Android 蓝牙锄大地扑克游戏项目在采用 Kotlin 技术路线时的开发环境安装、技术栈选型、依赖安装方式、代码质量工具配置，以及版本仓库中应纳入管理的关键文件清单。文档面向已有中大型工程经验、但首次开展 Android/Kotlin 项目开发的开发者，目标是在尽量少引入额外工具链的前提下，建立一套可复现、可协作、可扩展的工程基线。

Android 官方已明确采用 Kotlin-first 路线，Jetpack、官方示例、文档与培训资源均优先面向 Kotlin 设计；同时，Jetpack Compose 已成为 Android 现代原生 UI 的官方方案。([Android Developers](<https://developer.android.com/kotlin/first?utm_source=chatgpt.com>))

## 2\. IDE 与基础开发环境安装

### 2.1 必需软件

本项目建议安装并使用以下基础环境：

1. **Android Studio（稳定版）**
   Android 官方推荐使用 Android Studio 进行 Android 应用开发。Android Studio 安装向导会一并协助安装 Android SDK、平台工具和模拟器等核心组件。([Android Developers](<https://developer.android.com/studio/install?utm_source=chatgpt.com>))
2. **JDK**
   Android Studio 通常会随 IDE 一并提供适配的运行时环境。对于首次搭建 Android 项目，优先使用 Android Studio 自带的 JDK 管理方式即可，无需额外自行组织 Java 工具链。Android Studio 与 ONNX Runtime 的 Android 构建文档都要求本地具备 Java 运行环境。([Android Developers](<https://developer.android.com/studio/install?utm_source=chatgpt.com>))
3. **Android SDK**
   通过 Android Studio 安装：
   * Android SDK Platform
   * Android SDK Build-Tools
   * Platform-Tools
   * Emulator（如需模拟器）
4. **Git**
   用于版本控制与团队协作。

### 2.2 建议安装方式

建议直接从 Android Developers 官方页面下载安装 Android Studio 稳定版，并通过首次启动向导安装推荐 SDK 组件。Android 官方说明指出，Android 模拟器是测试应用的首选方式，但也可以使用真机部署和调试。([Android Developers](<https://developer.android.com/studio/install?utm_source=chatgpt.com>))

## 3\. 技术栈选型

## 3.1 核心语言与构建体系

### 3.1.1 Kotlin

本项目建议统一采用 **Kotlin**。Android 官方已明确提出 Kotlin-first 路线，Kotlin 与现有 Android 运行时及 Java 生态具备良好互操作性。([Android Developers](<https://developer.android.com/kotlin/first?utm_source=chatgpt.com>))

### 3.1.2 Gradle（Kotlin DSL）

构建系统建议采用：

* **Gradle Wrapper**
* **Kotlin DSL (**`build.gradle.kts`)
* **Version Catalog (**`gradle/libs.versions.toml`)

Gradle 官方文档说明，Version Catalog 用于集中定义依赖与版本；Android 官方也推荐在 Android 项目中将依赖与插件迁移到版本目录，以便获得集中管理与类型安全访问。([Gradle 文档](<https://docs.gradle.org/current/userguide/version_catalogs.html?utm_source=chatgpt.com>))

---

## 3.2 UI 技术栈

### 3.2.1 Jetpack Compose

UI 层建议采用 **Jetpack Compose**。Android 官方将 Compose 定义为现代 Android 原生 UI 工具包，适合声明式界面开发，并支持 Material 3。([Android Developers](<https://developer.android.com/compose?utm_source=chatgpt.com>))

### 3.2.2 Material 3

组件库建议采用 **Material 3**，以减少基础界面样式工作量，并与 Compose 配合使用。

### 3.2.3 Navigation Compose

如需页面跳转与导航管理，可引入 **Navigation Compose**。对于本项目而言，主要用于：

* 房间页
* 游戏页
* 结算页
* 设置页

---

## 3.3 并发与状态管理

### 3.3.1 Kotlin Coroutines

异步任务建议统一使用 **Kotlin Coroutines**。Android 官方指出，协程适合处理可能阻塞主线程的长任务，可用于提升可读性并减少回调复杂度。([Android Developers](<https://developer.android.com/kotlin/coroutines?utm_source=chatgpt.com>))

适用场景包括：

* 蓝牙收发线程
* AI 推理调用
* 定时器与倒计时
* Snapshot 广播
* 本地存储读写

### 3.3.2 StateFlow / SharedFlow

客户端控制层与 UI 层之间，建议使用 **StateFlow / SharedFlow** 作为状态分发机制。Android Developers 已提供专门文档说明这两类 Flow API 在 Android 中用于状态更新和多消费者事件分发。([Android Developers](<https://developer.android.com/kotlin/flow/stateflow-and-sharedflow?utm_source=chatgpt.com>))

---

## 3.4 数据序列化与协议层

### 3.4.1 kotlinx.serialization

前后端（主机/客户端）协议建议采用 **JSON + kotlinx.serialization** 实现。Kotlin 官方文档说明，应在 Gradle 中启用 `kotlin("plugin.serialization")` 插件，并引入 `kotlinx-serialization-json` 依赖。`@Serializable` 注解可自动生成序列化器，适合 DTO、快照对象及命令对象的编解码。([Kotlin](<https://kotlinlang.org/docs/serialization.html?utm_source=chatgpt.com>))

适用对象包括：

* `GameCommand`
* `PlayCardCommand`
* `PassCommand`
* `StateSnapshot`
* `PlayerSnapshot`
* `RoomSnapshot`

---

## 3.5 本地存储

### 3.5.1 DataStore

对于昵称、音效开关、规则开关、历史连接偏好等轻量配置，建议使用 **Jetpack DataStore**。Android 官方说明，DataStore 使用 Coroutines 与 Flow 进行异步、事务一致的数据存储，适合取代 SharedPreferences。([Android Developers](<https://developer.android.com/topic/libraries/architecture/datastore?utm_source=chatgpt.com>))

### 3.5.2 Room

若后续需要实现“对局记录、统计、回放索引”等结构化数据功能，再考虑引入 Room。DataStore 适合小型数据集，不适合复杂关系数据与部分更新。([Android Developers](<https://developer.android.com/topic/libraries/architecture/datastore?hl=zh-cn&utm_source=chatgpt.com>))

---

## 3.6 AI 推理组件

### 3.6.1 ONNX Runtime Android

AI 子系统建议采用 **ONNX Runtime Android**。ONNX Runtime 官方文档表明，其支持 Android 端部署与移动端推理，Android 对应的软件包为 `onnxruntime-android`。([ONNX Runtime](<https://onnxruntime.ai/docs/?utm_source=chatgpt.com>))

建议方案如下：

* 将模型文件放入 `app/src/main/assets/`
* 在 `ai/` 包下封装 `OnnxBot`
* 使用适配器模式将推理结果转换为 `PlayCardCommand`

当前阶段优先使用官方发布包，不建议自行从源码编译 ONNX Runtime。只有在需要特定执行提供器、裁剪体积或启用原生定制构建时，才转向源码构建方案。([ONNX Runtime](<https://onnxruntime.ai/docs/build/?utm_source=chatgpt.com>))

## 3.7 蓝牙通信

建议使用 **Android Bluetooth Classic API** 直接封装实现多人本地联机。由于本项目需要建立房主权威式主机、客户端发命令、服务端广播状态的持续连接通信模型，优先采用原生蓝牙套接字层封装，而非额外引入第三方蓝牙框架。Android 官方将 Android 设备连接与通信纳入标准平台能力，并通过 Android SDK 提供相应 API 与调试支持。([Android Developers](<https://developer.android.com/studio/install?utm_source=chatgpt.com>))

## 5\. 依赖管理方式说明

## 5.1 Gradle 是 Android 项目的主依赖管理器

Android/Kotlin 项目不存在单独对应 `package.json` 或 `requirements.txt` 的单一文件。依赖、插件、构建参数与打包配置由 Gradle 统一管理，核心文件包括：

* `settings.gradle.kts`
* 根级 `build.gradle.kts`
* 模块级 `app/build.gradle.kts`
* `gradle/libs.versions.toml`

其中，`libs.versions.toml` 用于集中管理依赖版本与插件版本，Gradle 与 Android Studio 对这一模式提供直接支持。([Gradle 文档](<https://docs.gradle.org/current/userguide/version_catalogs.html?utm_source=chatgpt.com>))

* `gradlew`

Gradle 官方建议通过 Wrapper 运行项目构建，因为 Wrapper 可确保团队与 CI 使用一致的 Gradle 版本。([Gradle 文档](<https://docs.gradle.org/?utm_source=chatgpt.com>))

### 6.3 依赖目录配置

* `gradle/libs.versions.toml`

### 6.4 应用模块

* `app/build.gradle.kts`
* `app/src/main/AndroidManifest.xml`
* 源代码与资源文件

### 6.5 质量工具配置

* `config/detekt/detekt.yml`
* `.editorconfig`
* `lint.xml`（如需自定义 Android Lint）
* Git hook 脚本（若仓库内保存脚本模板）

## 7\. 不应提交到版本仓库的文件

建议在 `.gitignore` 中排除以下内容：

* `.idea/`
* `.gradle/`
* `build/`
* `local.properties`
* 各类 `*.iml`
* 模拟器或本地缓存目录
* 签名文件：`*.jks`、`*.keystore`
* 私有密钥、token、证书
* 本地日志与临时文件

其中，`local.properties` 通常包含本地 Android SDK 路径，仅适用于开发者本机环境，不应纳入版本库。

## 9\. 推荐配置文件清单与职责说明

## 9.1 `settings.gradle.kts`

职责：

* 声明项目名称
* 声明模块
* 配置插件解析与仓库

建议保留内容：

* `rootProject.name`
* `include(":app")`
* `pluginManagement`
* `dependencyResolutionManagement`

---

## 9.2 根级 `build.gradle.kts`

职责：

* 声明根级插件别名
* 定义全局构建行为
* 预留统一构建逻辑入口

建议尽量保持轻量，不直接堆叠业务依赖。

---

## 9.3 `app/build.gradle.kts`

职责：

* 声明 Android 应用模块插件
* 配置 `namespace`、`compileSdk`、`minSdk`、`targetSdk`
* 启用 Compose
* 引入 Compose、Coroutines、Serialization、DataStore、ONNX Runtime 等依赖
* 配置测试、lint 与打包参数

---

## 9.4 `gradle/libs.versions.toml`

职责：

* 统一管理插件与依赖版本
* 提供类型安全访问器
* 避免版本字符串分散在多个构建脚本中

Android 官方建议在 `gradle` 目录下使用默认文件名 `libs.versions.toml`。([Android Developers](<https://developer.android.com/build/migrate-to-catalogs?hl=zh-cn&utm_source=chatgpt.com>))

---

## 9.5 `gradle.properties`

职责：

* 配置 Gradle 构建参数
* 声明 Kotlin 代码风格与增量构建选项
* 配置 AndroidX、JVM 参数等

常见内容包括：

```
org.gradle.jvmargs=-Xmx4g -Dfile.encoding=UTF-8
android.useAndroidX=true
kotlin.code.style=official
```

---

## 9.6 `.editorconfig`

职责：

* 统一 IDE 与格式化行为
* 配合 ktlint 使用
* 约束缩进、行宽、换行等基本规范

ktlint 与 Kotlin 风格治理通常会参考 `.editorconfig`。([Pinterest](<https://pinterest.github.io/ktlint/dev-snapshot/install/setup/?utm_source=chatgpt.com>))

---

## 9.7 `config/detekt/detekt.yml`

职责：

* 自定义 detekt 规则
* 调整问题严重级别
* 关闭不适用于当前项目的默认规则

---

## 9.8 `lint.xml`

职责：

* 自定义 Android Lint 检查级别
* 局部关闭误报规则
* 控制特定规则的警告/错误等级

Android 官方文档明确支持通过配置文件与 Gradle 配置 lint 选项。([Android Developers](<https://developer.android.com/studio/write/lint?utm_source=chatgpt.com>))

7. 优先实现最小可运行版本：本地单机规则引擎、蓝牙连通、命令收发、快照刷新，再逐步接入 AI。
