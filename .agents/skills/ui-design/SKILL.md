# UI Design

适用于 Android / Compose 中的复杂界面改造，尤其是容易出现重叠、压缩、错位、裁切的问题。

## 核心原则

1. 先读规范，再改界面。
2. 先做布局骨架，再调视觉细节。
3. 先查约束链，再改尺寸和偏移。
4. 用统一的尺寸规格管理布局，不要散落写常量。
5. 局部视觉偏移不能替代主布局。

## 工作流程

1. 阅读相关设计、字体、布局适配文档，确认当前阶段目标。
2. 先拆出稳定区域，再安排组件内容。
3. 建立统一的 `LayoutSpec` / `Dimens`，集中管理尺寸、间距和分级。
4. 先保证可见性和层级关系，再处理阴影、光效、动画。
5. 每轮修改后记录根因、已证伪方向和下一步验证点，避免反复试错。

## 布局规则

1. 复杂页面优先使用分区布局，不要把所有元素堆在一个大容器里靠 `offset` 摆放。
2. 横屏页面必须同时考虑宽度和高度，不能只按宽度分档。
3. 关键内容区必须有保底空间，不能被次要区域挤压掉。
4. 尺寸适配优先通过约束范围实现，如 `widthIn`、`heightIn`、`sizeIn`，而不是反复硬调 `dp`。
5. 当元素密集排列时，应把“位置计算”和“视觉渲染”分开。

## 排查顺序

遇到重叠、压缩、错位、裁切时，按下面顺序检查：

1. 父容器是否有限高、限宽或 `clip`。
2. 是否有多层 `padding` / inset 叠加吃掉空间。
3. 当前改动是否真正改变了约束，而不只是改了表面参数。
4. 先找真正生效的上层约束，不要先认定局部参数就是主控开关。
5. 用最小实验验证哪一层真的起作用；如果局部尺寸、偏移、对齐改动无效，应继续向父级容器和布局骨架上溯。
6. 关键区域是否有最小可用空间。
7. 问题是否属于布局结构，而不是组件绘制本身。

## 常见误区

1. 看到问题就先调几个 `dp`。
2. 用 `offset` 解决结构性布局问题。
3. 改了对齐方式，却没有确认容器内部是否真的有可移动空间。
4. 把安全区问题和布局问题混在一起处理。
5. 把局部参数误当成主控开关，而没有继续检查更上层的高度、宽度和分区约束。
6. 在没有明确根因时连续多轮试错式修改。

## 实施要求

1. 优先做最小正确改动。
2. 结构稳定前，不要过早投入大量视觉细节优化。
3. 如果一个问题连续两轮没有收敛，应停下来重查布局模型，而不是继续补丁式修改。
